# 要件定義書 フェーズ7：X（Twitter）コンテンツ取得

## 概要

Slack メッセージに X（Twitter）の投稿 URL が含まれている場合、外部の X フェッチエンドポイント（`X_FETCH_ENDPOINT`）経由でツイート内容を取得し、Notion・Upstash Vector に保存する。あわせて、外部コンテンツに起因するプロンプトインジェクション攻撃への対策を講じる。

---

## 背景・目的

フェーズ3で URL コンテンツの自動取得が実装されたが、X（Twitter）は認証が必要なため `extractor.ts` のスキップドメインに登録されており、X の投稿は収集対象外となっている。本フェーズでは外部エンドポイントを活用して X コンテンツを収集対象に加え、ナレッジベースの情報量を拡充する。

また、外部から取得したテキストには悪意あるプロンプト指示が含まれる可能性があるため、Gemini に渡す前に無害化処理を行う。

---

## 機能要件

### FR-1: X URL 検出

- Slack の生テキストから `x.com`・`twitter.com`・`t.co` ドメインの URL を抽出する
- 1メッセージにつき最大 2件を処理する（負荷対策）
- 現在スキップ対象になっている X URL を専用フローへ切り替える

### FR-2: ツイート内容取得

- `X_FETCH_ENDPOINT` に対して POST リクエストを送信し、ツイート内容を取得する
- リクエスト形式: リクエストボディに `{"url": "<X の投稿 URL>"}` を含める
- レスポンスの `success` が `false` または取得失敗の場合は処理をスキップし、既存フローに影響させない
- タイムアウト: 10 秒

### FR-3: Notion・Vector への保存

- 取得したツイート内容を既存の `fullText` に追記し、`processSingleMessage()` の通常フローで Notion・Upstash Vector に保存する
- Notion ページへの追記フォーマット:

  ```
  [X投稿: @{screen_name}]
  {ツイート本文}

  👍 いいね: {likes}  🔁 RT: {retweets}
  投稿日時: {created_at}
  URL: {tweet_url}
  ```

### FR-4: プロンプトインジェクション対策

外部から取得したテキストは Gemini に渡す前に必ず無害化処理を行う。

| 検知パターン | 対応 |
|---|---|
| `ignore previous instructions` などの英語指示文 | 該当箇所を `[removed]` に置換 |
| `system:` / `assistant:` / `user:` で始まる行 | 同上 |
| 「あなたは今から〜」などの日本語ロール指定 | 同上 |
| 10,000 文字超のテキスト | 先頭 3,000 文字に切り詰め |

---

## 非機能要件

- X コンテンツ取得の失敗（非公開・削除済み・タイムアウト等）は例外を握りつぶし、既存のメッセージ保存フローを継続する
- プロンプトインジェクション対策は X コンテンツに限らず、今後 URL 取得コンテンツなど外部テキスト全般に適用できる設計とする
- `X_FETCH_ENDPOINT` は環境変数で管理し、ソースコードに直書きしない

---

## スコープ外：フェーズ7以前の過去データについて

本フェーズはフェーズ7デプロイ以降に投稿された X URL を対象とする。**過去に保存済みの Notion ページへのバックフィルは対象外**とする。

### 理由

Slack の生テキストに含まれる X URL（`<https://x.com/...|テキスト>` 形式）は、保存処理の中で `cleanText()` によって URL 部分が除去され、Notion にはリンクテキストのみが保存されている。そのため、既存の Notion ページから元の X URL を復元する手段がなく、バックフィルを行うには Slack のチャンネル履歴を再スキャンする専用スクリプトが必要になる。対応コストに対してメリットが小さいため、今回は見送る。

---

| 変数名 | 用途 |
|-------|------|
| `X_FETCH_ENDPOINT` | X コンテンツ取得エンドポイントの URL |
